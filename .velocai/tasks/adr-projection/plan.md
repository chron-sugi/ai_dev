# Plan: Dual-target ADR projection (Copilot + Claude), per-domain, update-in-place

## Goal

Implement the projection generator that compiles ADR frontmatter from `docs/adrs/` into two tool-facing surfaces per domain: `.github/instructions/<domain>.instructions.md` (VS Code Copilot, `applyTo:` glob frontmatter) and `.claude/rules/<domain>.md` (Claude Code, `paths:` glob frontmatter). Generation is idempotent: files are created when absent, rewritten in place when the derived content changes, left byte-identical when nothing changed, and deleted only when they carry the generated notice and no longer correspond to a live domain. One internal model feeds both targets so the two surfaces can never drift from each other or from the ADR store.

## Non-goals

- No `adr-lint` as a standalone tool — this plan includes only the load-time validation the generator itself needs (fail fast on malformed frontmatter). A full standalone linter with status-transition checks is a separate task.
- No CI workflow — the repo has no `.github/workflows/` yet. This plan delivers the `just project-check` recipe CI will call; wiring CI is a follow-up.
- No lint/golden-file projection channels (`projection: lint`) — ADRs marked `lint` are listed in the index but their enforcement config is Phase-3 work per the domain PLAN.md.
- No probe suite, no token-budget dashboard.
- No changes to any ADR body or to the accepted ADR-0001 (immutable).

## Approach

**One model, two renderers.** The generator parses every `docs/adrs/*.md` frontmatter block into an `AdrRecord`, filters to projectable records (status in the include set, `projection` includes the instructions channel, non-empty `rule`), clusters them by `domain:` (ADR-0007 — never one file per ADR), and derives a **desired-outputs map**: `{target_path: rendered_content}` covering both tool surfaces plus the universal always-on surface. Rendering both targets from the same clustered model is what makes "generated by domain, updated if already exists" trivial and safe: update-in-place is just "write desired content if the file differs," and staleness cleanup is "delete any notice-bearing file in the target directories that is not in the map." ADR-0006 explicitly requires this single desired-outputs shape so one drift guard covers every target.

**Dialect differences are confined to frontmatter.** Both per-domain files share an identical generated body (one bullet per rule, `(ADR-NNNN)` citation, pointer to read the full ADR before touching in-scope files). The Copilot renderer emits `applyTo:` as a comma-joined string; the Claude renderer emits `paths:` as a YAML list. Universal rules (`scope: "**"` or repo-wide) go to a sentinel-marked block in `.github/copilot-instructions.md` (hand-authored prose may surround it, per ADR-0006 sentinels) and to `.claude/rules/universal.md` with no `paths:` frontmatter — which Claude Code always loads, giving the same effect without touching the hand-authored `CLAUDE.md`/`AGENTS.md`.

**Determinism and idempotency are load-bearing.** No timestamps in output; domains, rules (by ADR id), and globs (sorted, deduplicated) all have stable order; writes happen only on content change. This makes `--check` a pure recompute-and-diff, makes supersession a clean in-file diff (ADR-0007's stated payoff), and keeps re-runs from churning git history. Every fully generated file begins with the notice comment `<!-- generated by velocai projection from docs/adrs/ — do not edit; edit the source ADR and run `just project` -->`; Claude strips HTML comments before injection so the notice costs zero context tokens on that surface. Deletion authority is gated on that exact notice string (ADR-0006: only notice-bearing files may be auto-deleted).

**Stdlib only, entered only through just.** ADR-0012 requires framework scripting to run from a bare interpreter, so frontmatter parsing is a small hand-rolled parser for the YAML subset ADR frontmatter actually uses (scalar strings, quoted strings, string lists) — no PyYAML import. The ADR corpus is machine-authored via `/write-adr`, so a strict subset parser that fails loudly on anything unexpected is safer than a permissive full-YAML dependency. Entry points are `just project` and `just project-check` (names fixed by ADR-0006's Decision text); the recipes invoke `py -3 -m app.projection` — pipeline steps are never run as bare scripts (ADR-0008).

- **Integration:** end-to-end tmp-repo test — synthetic `docs/adrs/` in, both target trees out, then: edit a rule → both files update; delete a domain's ADR → both files removed; add a hand-authored `.github/instructions/manual.instructions.md` → untouched.
- **E2E/manual:** run against the real repo (step 7); open one generated file in VS Code and confirm Copilot picks up `applyTo`; start a Claude Code session and confirm a path-scoped rule loads when reading a matching file.
- **Manual negative:** hand-edit a generated file, run `just project-check`, confirm non-zero exit and readable drift report.

## Rollback / abort

Fully revertable: the generator is a new package plus new generated files — `git revert` of the commits removes everything; no migrations, no schema changes, no existing-file mutations outside the two new justfile recipes and the sentinel block in a previously nonexistent `.github/copilot-instructions.md`. Partial abort is also clean because steps 1–5 are committable independently and inert until the just recipes (step 6) exist.

## Open questions

All resolved by recommendation below (autonomous session — user can override any of these before implementation starts):

- **OQ-1, status gate.** The domain PLAN.md says project `accepted` only, but 20 of 21 ADRs sit at `proposed` while being treated as operative (AGENTS.md cites them as binding). **Recommendation: include `accepted` + `proposed`, exclude `superseded`/`deprecated`,** and treat flipping operative ADRs to `accepted` as a separate ratification pass; when that lands, tightening the include set is a one-line change.
- **OQ-2, ADR-0001's legacy `projection:` path-list.** ADR-0001 is `accepted` → immutable, so the generator normalizes the legacy form (step 1) rather than the ADR being edited. If the frontmatter schema is later formalized, a superseding ADR can retire the legacy form.
- **OQ-3, script location.** ADR-0006/0007 prose mentions `scripts/project-adrs` / `scripts/adr-lint`, but those are Decision-prose mentions in *proposed* ADRs, not `rule:` fields, and AGENTS.md places application code in `src/app/`. **Recommendation: `src/app/projection/` module invoked via just** — since ADR-0008 makes recipes the only entry point, the module path is an implementation detail.
- **OQ-4, universal Claude surface.** **Recommendation: `.claude/rules/universal.md` (no `paths:`, always loaded)** rather than a sentinel block in `CLAUDE.md`/`AGENTS.md`, keeping the hand-authored root file untouched by the pipeline.
- **OQ-5, token budget gate.** PLAN.md Phase 2 wants a build-failing budget on the always-loaded surface (~2,000 tokens). Worth doing, but it needs a token-estimation decision; **recommendation: land as a fast follow to this plan** with a chars/4 heuristic and a warning-then-fail threshold.
